## 1. AI Transform Mode ‚Äî Voice-Driven Text Rewriting

**Commit:** `b99045f`

### User-facing (help center)

Select any text in any app, press your recording hotkey, and speak an instruction like "make this more formal" or "translate to Spanish." Yapper replaces the selected text with an AI-rewritten version. The result streams token-by-token into an expanded overlay card where you can review it. Once streaming completes, the original selection is deleted and replaced with the transformed result. Press Escape to dismiss without replacing.

### Requirements

- A configured AI provider (Gemini, OpenAI, Anthropic, or xAI) with API key
- Accessibility permission (prompted on first launch; required to use the app)
- Selection capped at 10,000 characters
- Secure text fields (passwords) are detected and skipped ‚Äî falls back to normal dictation

### Keyboard behavior

| Key | Action |
|-----|--------|
| Escape | Cancel streaming / dismiss result card (no replacement) |
| Recording hotkey | Blocked while transform is active |

### Technical details

- **New file:** `AccessibilityReader.swift` (112 lines) ‚Äî reads selected text via AXUIElement APIs with 200ms timeout; detects and skips secure text fields
- **New file:** `docs/analysis/features/ai-transform.md` (202 lines) ‚Äî internal feature documentation
- `InteractionMode.aiTransform(selectedText:)` enum branches the recording flow
- `LLMProvider.enhanceStream()` protocol method ‚Äî all 4 providers implement native SSE streaming
- `LLMService.transform()` / `transformStream()` ‚Äî dedicated transform methods with hardcoded prompt instructing the LLM to return only the transformed result with no commentary
- `TextInjector.deleteSelection()` ‚Äî simulates Delete key (keyCode 0x33) to clear the selected text before injecting the transformed result
- Cursor-anchored positioning via `positionAtCursorAnchored()` ‚Äî stores initial cursor position and preserves card placement when transitioning from pill to expanded card
- New `RecordingState` cases: `.aiTransforming`, `.aiTransformResult`
- New `OverlayDisplayState` cases: `.aiRecording` (purple), `.aiTransforming` (purple), `.aiTransformResult(hasError:)` (green/orange)
- `resolveProvider()` internal refactor extracts duplicated provider resolution logic across LLMService

### Settings

None added. Uses existing AI provider configuration.

### Permissions

Accessibility (AXUIElement) ‚Äî prompted on first launch and required to use the app. The prompt persists until the user grants permission.

## 2. AI Q&A Voice Assistant ("Hey Yapper")

**Commits:** `a4073d1` + `b536d00` (fuzzy matching folded in)

### User-facing (help center ‚Äî should be extensively documented)

Say "Hey Yapper" followed by a question during any recording to get an AI-generated answer in a floating panel. For example: "Hey Yapper, what's the capital of Japan?" The feature recognizes 35+ pronunciation variants including "hey upper," "hay yapper," "hey rapper," and others to handle common speech-to-text misrecognitions.

The feature is briefly described in Settings > AI, but the help center should be the primary reference covering:

- How to activate (say "Hey Yapper" + question during recording)
- Example questions and use cases
- Full list of recognized wake-phrase variants
- What happens when no AI provider is configured (treated as normal dictation)
- Troubleshooting: if "Hey Yapper" isn't recognized, try speaking clearly with a pause after "Hey Yapper"

### Requirements

Configured AI provider with API key. No additional permissions beyond existing microphone access.

### Keyboard behavior

| Key | Action |
|-----|--------|
| Escape | Cancel AI request / dismiss Q&A panel |

### UI flow

Recording pill ‚Üí purple "Thinking..." pill ‚Üí expanded card with streaming answer ‚Üí green "Answer Ready". Question shown in dimmed header above the answer. Copy button available when complete.

### Technical details

- `detectHeyYapper(in:)` ‚Äî iterates 35 regex patterns organized by category:
  - Canonical: `hey\s+yapper`
  - Vowel/consonant swaps: yaper, yappar, yappor, yappur, yepper, yeper, yipper, yopper, yupper
  - Missing leading Y: apper, upper
  - Y‚Üíother consonant swaps: rapper, japper, napper, dapper, tapper, zapper, jabber, yabber
  - Plural/suffix drift: yappers, "yap per", "yap her", "yap"
  - "Hey" variants: hay, "hey a yapper", "a yapper", hei
  - Merged tokens: heyyapper
- `InteractionMode.aiQA(question:)` enum case
- `LLMService.qaStream(question:)` ‚Äî system prompt: "You are a helpful voice assistant called Yapper..."
- Post-transcription routing priority: AI Transform ‚Üí AI Q&A ‚Üí Normal Dictation
- Shared streaming infrastructure with Transform mode:
  - `startAIResponseStreaming(recordingState:)`
  - `appendAIResponseToken(_:)`
  - `completeAIResponseStream(recordingState:)`
  - `failAIResponseStream(_:resultState:errorPrefix:)`
- New `RecordingState` cases: `.aiQA`, `.aiQAResult`
- New `OverlayDisplayState` cases: `.aiQA` (purple), `.aiQAResult(hasError:)` (green/orange)
- Q&A card UI: question display bar with `?` icon (max 3 lines), speech-bubble icon, dynamic header "Thinking..." / "Answer Ready", auto-scrolling answer body
- Copy button copies raw answer text; Escape dismisses

### Renamed state properties (from Transform-specific to shared)

| Old name | New name |
|----------|----------|
| `AppState.transformResult` | `AppState.aiResponseText` |
| `AppState.transformStreamError` | `AppState.aiResponseError` |
| `AppState.isTransformStreaming` | `AppState.isAIResponseStreaming` |
| `AppState.dismissTransformResult()` | `AppState.dismissAIResponse()` |
| `OverlayDisplayState.transformResultCard` | `OverlayDisplayState.aiResponseCard` |
| `OverlayDisplayState.isTransformCard` | `OverlayDisplayState.isAIResponseCard` |

### Settings

None added. Activates automatically when an AI provider is configured.

## 3. Microphone Selection

**Commit:** `0031077`

### User-facing (help center)

Choose which microphone Yapper uses from Settings > Transcription > Input Device. A dropdown lists all connected audio devices plus "System Default." A live audio level meter below the picker shows real-time input levels with green/yellow/red color coding. If your selected microphone is disconnected, Yapper automatically falls back to the system default and displays a warning banner in Settings.

### Hot-plug support

New devices are detected instantly when plugged in; removed devices trigger automatic fallback.

### New settings

- **Input Device** dropdown picker in Settings > Transcription
- **Live audio level meter** (4px bar, green/yellow/red thresholds at 0.5‚Äì0.8)
- **UserDefaults key:** `selectedMicrophoneUID` (String?, nil = system default)

### Technical details

- **New file:** `AudioInputDevice.swift` ‚Äî model struct with dual identifiers: volatile `audioDeviceID` (UInt32) + stable `uid` (String); implements Identifiable, Hashable, Sendable
- **New file:** `AudioDeviceManager.swift` ‚Äî CoreAudio HAL enumeration, property listeners for hot-plug, UID-to-runtime-ID resolution, level metering via temporary AVAudioEngine instances
- **New file:** `MicrophoneLevelView.swift` ‚Äî SwiftUI level meter with 80ms easeOut animation, session-based monitoring lifecycle
- `AudioRecorder.startRecording()` signature changed:
  - Old: `func startRecording(_ callback: @escaping (Float) -> Void)`
  - New: `func startRecording(deviceID: AudioDeviceID?, callback: @escaping (Float) -> Void) -> Bool`
- `AudioRecorder.setInputDevice(_:)` ‚Äî configures AudioUnit via CoreAudio HAL (uninitialize ‚Üí set device ‚Üí reinitialize)
- `AppState` additions: `selectedMicrophoneUID`, `availableInputDevices`, `audioDeviceManager`, `microphoneFellBack`
- Settings level monitoring stops during active recording to prevent engine conflicts
- Implementation note: macOS has no AVAudioSession (iOS-only), so device selection uses CoreAudio HAL directly on the AudioUnit backing AVAudioEngine's inputNode

### Permissions

None new. Uses existing microphone access.

## 4. Markdown Rendering in AI Responses

**Commit:** `6a9658d`

### User-facing (help center)

AI responses from both Transform and Q&A modes now display with full Markdown formatting ‚Äî headings, bold, italic, code blocks, lists, tables, and blockquotes. The copy button copies the raw Markdown text.

### Technical details

- Custom theme typography:
  - Base text: 13pt, 90% opacity
  - Inline code: monospace, purple-tinted, 0.9em, subtle background

  - Blockquotes: left-border style, dimmed italic
  - Tables: subtle borders, tight cell padding
- Streaming-compatible: Markdown re-parses on every token append
- Auto-scroll preserved via ScrollViewReader with onChange observer
- Text selection enabled on rendered Markdown content

## 5. Settings Window Redesign + AI Mode Changes

**Commits:** `ae668d5` + `56b4dd3`

### User-facing (help center)

The Settings window has been completely redesigned:

- **Sidebar navigation** ‚Äî modern macOS System Settings style with `NavigationSplitView` (replaces horizontal segmented tabs)
- **Larger window** ‚Äî 750x520, resizable (min 650x400), 12pt continuous corner radius
- **"General" renamed to "Preferences"**
- **"AI Enhancement" renamed to "AI"** ‚Äî now explains the two user-initiated modes (Transform and Q&A) with icons and descriptions
- **Permissions banner** ‚Äî orange gradient banner at the top when microphone or accessibility permissions are missing, with contextual messages and "Grant Access" button
  - Microphone: opens `x-apple.systempreferences:com.apple.preference.security?Privacy_Microphone`

- Old permissions section removed from Preferences pane

### Breaking behavior change ‚Äî auto-enhancement removed

Transcriptions are no longer automatically enhanced via AI after recording. The "Enable AI Enhancement" toggle and custom prompt editor are gone. AI is now strictly user-initiated:

- **Transform:** select text + press hotkey + speak instruction
- **Q&A:** say "Hey Yapper" + question

Users upgrading from 1.x who relied on auto-enhancement will need to adapt to the new workflow. There is no equivalent automatic post-processing in 2.0.

- `LLMService.enhance(text:)`, `setPrompt()`, `getPrompt()`, `defaultPrompt` removed
- `Notifications.aiPromptChanged` removed
- Transcription routing simplified: `!supportsStreaming || isTransformMode` (removed `useEnhancement` condition)

## 7. Free Trial Licensing (7-day)

**Commit:** `6e62565`

### User-facing (help center)

New users get a 7-day free trial automatically on first launch. A toast confirms "Free trial started ¬∑ 7 days."

- **During trial:** Settings > Preferences shows an orange trial badge with remaining days and a "Buy a license" link
- **Mid-trial purchase:** When a valid license key is entered, the trial badge disappears immediately and transitions to a fully licensed state
- **After expiry:** A modal appears with "Your Free Trial Has Ended," offering "Buy Yapper" (links to yapper.to) or "I Already Have a License" to enter a key

### New settings UI

- Trial badge in Settings > Preferences (orange clock icon + days remaining)
- Inline "Buy a license" link
- Trial-expired modal with "Buy Yapper" button and "I Already Have a License" button
- Toast notification: icon `clock.badge.checkmark.fill`, text "Free trial started ¬∑ 7 days" (shown only on first trial launch)

### New UserDefaults key

- `com.yapper.trial.welcomed` (Boolean) ‚Äî tracks whether welcome toast was shown

### URL change

Purchase link now points to `https://yapper.to/` (previously pointed to Polar storefront with conditional sandbox/production branching).

### Technical details ‚Äî TrialService

**New file:** `TrialService.swift` (175 lines) ‚Äî singleton `TrialService.shared`:

- **Storage:** macOS Keychain with opaque identifiers:
  - Service: `"app.persistence.layer"`
  - Account: `"session.token.v1"` (payload), `"install.marker.v1"` (tombstone)
  - Accessibility: `kSecAttrAccessibleAfterFirstUnlock`
- **Payload:** JSON-encoded `TrialPayload` struct: `startedAt`, `lastSeenAt`, `signature`
- **Security:**
  - HMAC-SHA256 signing over `"yapper.trial.v1:{startedAt}:{bundleId}"` with 32-byte embedded key
  - Tamper detection: invalid HMAC immediately expires trial
  - Tombstone pattern: empty Keychain item survives payload deletion, preventing fresh-start bypasses
  - Clock rollback detection: if `now < lastSeenAt`, trial expires
- **Decision table (`checkOrStartTrial()`):**
  - No payload + no tombstone ‚Üí first launch ‚Üí `.active(7)`
  - No payload + tombstone ‚Üí deletion attack ‚Üí `.expired`
  - Bad HMAC ‚Üí tampered ‚Üí `.expired`
  - Clock rollback ‚Üí `.expired`
  - Elapsed >= 7 days ‚Üí `.expired`
  - Elapsed < 7 days ‚Üí `.active(remaining)`
  - Keychain unavailable on first launch ‚Üí grant trial, retry on next launch

### LicenseState changes

- New enum cases: `.trialActive(daysRemaining: Int)`, `.trialExpired`
- `canUseApp` computed property returns `true` for both `.valid` and `.trialActive`
- New method: `setTrialState(_ state: LicenseState)` (MainActor-isolated)

### AppState changes

- New properties: `isInTrial: Bool`, `trialDaysRemaining: Int?`
- New methods: `updateTrialState(daysRemaining:)`, `markTrialExpired()`
- `updateLicenseState(isValid:...)` clears trial state when a real license is activated

### AppDelegate flow

1. After Polar license check fails ‚Üí `TrialService.shared.checkOrStartTrial()`
2. `.active` ‚Üí sets LicenseService and AppState, calls `touchLastSeen()`, proceeds with normal app setup
3. Shows welcome toast only on first trial launch (keyed by `com.yapper.trial.welcomed` UserDefault)
4. `.expired` ‚Üí sets state and shows license activation modal
